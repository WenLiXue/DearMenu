## 1. 概述

### 1.1 功能目标

实现家庭点餐协作流程：老婆选择菜品 → 生成订单 → 推送给老公端 → 老公处理任务 → 完成反馈通知老婆。

### 1.2 现有架构分析

项目已有以下相关组件可复用：

| 组件 | 说明 | 复用方式 |
|------|------|----------|
| `OrderHistory` 模型 | 承担订单功能 | 扩展字段 |
| `TaskStatus` 枚举 | PENDING/COOKING/COMPLETED | 直接使用 |
| `Notification` 系统 | 通知推送 | 扩展类型 |
| Husband 路由 | 老公端任务 API | 扩展端点 |

## 2. 数据库模型设计

### 2.1 现有模型扩展

**结论**: 复用现有 `OrderHistory` 表，无需新建 orders 表。

```python
# models.py 现有结构（无需修改）
class OrderHistory(Base):
    __tablename__ = "order_history"

    id = Column(UUID(as_uuid=True), primary_key=True, default=uuid.uuid4)
    user_id = Column(UUID(as_uuid=True), ForeignKey("users.id", ondelete="CASCADE"), nullable=False)  # 点餐者（老婆）
    family_id = Column(UUID(as_uuid=True), ForeignKey("families.id", ondelete="CASCADE"), nullable=False)
    dish_id = Column(UUID(as_uuid=True), ForeignKey("dishes.id", ondelete="CASCADE"), nullable=False)
    status = Column(Enum(TaskStatus), default=TaskStatus.PENDING, nullable=False)
    created_at = Column(DateTime(timezone=True), server_default=func.now())
    cooked_at = Column(DateTime(timezone=True), nullable=True)  # 完成时间
```

**建议扩展字段**（可选，未来可加）:

```python
# 扩展字段（v2 版本考虑）
notes = Column(Text, nullable=True)           # 点餐备注（少盐、不要辣等）
quantity = Column(Integer, default=1)        # 数量
```

## 3. 后端 API 设计

### 3.1 API 列表

| 方法 | 路径 | 说明 | 权限 |
|------|------|------|------|
| POST | `/api/orders` | 创建订单（老婆点餐） | wife |
| GET | `/api/orders` | 获取订单列表 | 全部 |
| GET | `/api/orders/{id}` | 获取订单详情 | 全部 |
| PATCH | `/api/orders/{id}/status` | 更新订单状态 | husband |
| GET | `/api/orders/pending` | 获取待处理订单 | husband |
| POST | `/api/orders/{id}/notify` | 主动通知（提醒） | wife |

### 3.2 详细接口设计

#### 3.2.1 创建订单 POST /api/orders

**请求体**:

```json
{
  "dish_id": "uuid",
  "notes": "少盐，不要辣"  // 可选
}
```

**响应**:

```json
{
  "code": 200,
  "message": "订单已创建",
  "data": {
    "id": "uuid",
    "dish_id": "uuid",
    "status": "pending",
    "created_at": "2026-04-07T12:00:00Z"
  }
}
```

**业务逻辑**:

1. 验证 dish_id 属于当前 family
2. 创建 OrderHistory 记录
3. **自动推送通知给老公端**：创建 Notification，type="task"

#### 3.2.2 更新订单状态 PATCH /api/orders/{id}/status

**请求体**:

```json
{
  "status": "cooking" | "completed"
}
```

**响应**:

```json
{
  "code": 200,
  "message": "状态已更新",
  "data": {
    "id": "uuid",
    "status": "cooking",
    "cooked_at": null  // completed 时设置为当前时间
  }
}
```

**业务逻辑**:

1. 老公端更新状态时，自动通知老婆端
   - `cooking` → 发送 Notification 告知"老公开始做了"
   - `completed` → 发送 Notification 告知"菜品已做好"

#### 3.2.3 获取待处理订单 GET /api/orders/pending

**响应**:

```json
{
  "code": 200,
  "message": "success",
  "data": [
    {
      "id": "uuid",
      "dish_id": "uuid",
      "dish_name": "红烧肉",
      "status": "pending",
      "created_at": "2026-04-07T12:00:00Z",
      "notes": "少盐"
    }
  ]
}
```

### 4.3 状态同步

**WebSocket 方案**（推荐，实时性高）:

```typescript
// hooks/useOrderSync.ts
const useOrderSync = () => {
  useEffect(() => {
    // 监听订单状态变化
    socket.on('order:updated', (order) => {
      // 更新本地状态
    });

    // 监听新订单通知
    socket.on('order:new', (order) => {
      // 显示通知
    });

    return () => {
      socket.off('order:updated');
      socket.off('order:new');
    };
  }, []);
};
```

## 8. 风险与注意事项

1. **并发问题**: 同一菜品可能被多人同时点餐，需要做好数据一致性
2. **通知可靠性**: WebSocket 断线重连机制需要完善
3. **状态一致性**: 前后端状态需要保持同步，避免状态冲突
4. **取消订单**: 需要考虑老公已经开始制作后老婆取消的场景（建议不允许）
